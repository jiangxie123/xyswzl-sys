# 校园失物招领系统 · 演示顺序

> 适用场景：**课程设计答辩 / 验收演示 / 给别人快速展示**。
> 目标：**在 5~10 分钟内，让观众看到项目核心价值和亮点**。

---

## 前置条件（演示前 1 分钟检查）

- [ ] MySQL 已启动，`xyswzl_db` 数据库存在且已初始化（执行过 `init_database.sql`）
- [ ] 项目已启动：`http://localhost:8080/` 能打开登录页
- [ ] 记住两个账号：`admin / 123456`（超级管理员），`student01 / 123456`（学生）
- [ ] 浏览器打开一个**无痕窗口**或**另一个浏览器**用于模拟不同角色同时登录（避免同一浏览器 Token 覆盖）

---

## 方案 A：5 分钟快速演示（推荐答辩用）

### 第 1 步 · 登录页展示（30 秒）

- 打开 `http://localhost:8080/` → 自动跳 `/login`
- 展示：**验证码**（前端 Canvas 生成，点击图片刷新）、**密码加密传输**（演示时不用打开 DevTools，但可以提一下）
- 用 `admin / 123456` 登录 → 进入主界面

### 第 2 步 · 管理员后台：用户管理（1 分钟）

- 左侧菜单 → `用户管理`
- 展示：4 个预置账号（admin / manager / student01 / student02）
- **亮点演示**：点击「编辑」普通管理员 → 演示「普通管理员不能修改其他管理员」（如果想用普通管理员操作，再开一个窗口用 `manager / 123456` 登录尝试，会被权限拦截）
- 演示新增一个测试学生（随意填写用户名 + 密码 + 学号）

### 第 3 步 · 管理员后台：物品审核（1 分钟）

- 切换到 `物品审核`
- 展示：初始数据里的 3 条物品信息（"丢失学生证"、"丢失黑色手机"、"捡到一件蓝色外套"），它们默认 `audit_status=1` 已审核，所以审核列表可能为空
- **演示发布-审核流程闭环**：先用学生账号 `student01 / 123456` 在另一个窗口发布一个新物品（标题比如"测试-丢失校园卡"，类型选「失物」）→ 再回到管理员窗口刷新「物品审核」→ 看到新物品 → 点「通过」或「驳回」

### 第 4 步 · 学生视角：列表 + 详情 + 留言（1.5 分钟）

- 在学生窗口 `student01`，点击顶部「失物招领」→ 进入物品列表
- 展示：**分类筛选**（左侧或顶部 Tab）、**类型切换**（失物 / 拾物）、**关键词搜索**（搜"手机" / "外套"）
- 点击任一条进入物品详情 → 展示图片、描述、联系方式、**留言区**
- 发表一条留言 → 刷新列表确认出现

### 第 5 步 · 认领流程 + 我的发布（1 分钟）

- 在物品详情页，点「已找回/已认领」按钮（演示状态变更）
- `我的发布` 页面查看状态变化（`待认领 → 已认领`）
- 展示：只能操作自己的物品（归属校验生效）

### 第 6 步 · 管理员后台：操作日志 + 清理（30 秒）

- 管理员窗口 → `操作日志`
- 展示：刚才审核、发布、修改状态的操作都被记录了
- **亮点**：超级管理员右上角「清理 30 天前日志」按钮 → 演示日志清理 API（普通管理员看不到）

### 第 7 步 · 小结（30 秒）

一句话收尾：**"这是一个完整的校园失物招领系统，包含用户注册登录、信息发布、审核、留言、认领、操作审计 —— 使用 Spring Boot + Vue 3 + MySQL 实现，采用 JWT 鉴权，角色三级权限划分。"**

## 方案 B：10 分钟完整演示（推荐正式验收）

### 阶段 1 · 系统概览（1 分钟）
- 打开 `README.md` 或直接口述：项目做什么、技术栈（Spring Boot 4.1 + Vue 3 + MySQL 8 + JWT）

### 阶段 2 · 账号体系与角色演示（2 分钟）
- 演示 `admin / 123456` → 拥有所有菜单
- 演示 `manager / 123456` → 不能删除用户，不能新增其他管理员
- 演示 `student01 / 123456` → 只能看列表/详情/发布，看不到管理员菜单
- **亮点**：同一个 URL，不同角色看到的菜单完全不同（前端路由守卫 + 后端接口权限双重拦截）

### 阶段 3 · 注册新用户（1 分钟）
- 登录页 → 切换到「注册」Tab
- 填一个新账号：`testuser01 / 123456 / 真实姓名 / 学号 / 学院`
- 注册成功 → 自动跳登录 → 用新账号登录 → 进入学生界面
- 管理员窗口 → `用户管理` → 刷新列表，新用户出现在最后

### 阶段 4 · 发布一条新信息（2 分钟）
- 用 `testuser01` 发布：类型=拾物，分类=证件类，标题=「捡到一张图书卡」，描述=「图书馆一楼捡到」，上传一张图片，填写联系方式
- 提交成功 → 提示"发布成功，等待审核"
- `我的发布` 页面查看自己的发布 → 状态：待审核

### 阶段 5 · 审核信息（1 分钟）
- 切到 `admin` → `物品审核` → 看到刚才那条待审核信息
- 点击「通过」→ 物品公开
- 演示「驳回」：再发一条测试 → 驳回 → 列表里不显示（或在已驳回里显示）

### 阶段 6 · 留言与认领（1 分钟）
- 用 `student02 / 123456`（另一个窗口）访问刚审核通过的物品 → 留言
- 切回发布者 `testuser01` → 看到留言 → 状态改为「已认领/找回」
- 刷新列表 → 物品状态已更新

### 阶段 7 · 管理员功能展示（1 分钟）
- 分类管理：新增一个分类「电子设备-配件」，修改排序号，删除
- 操作日志：按模块筛选（USER / ITEM / CATEGORY），展示完整操作轨迹
- 清理日志按钮演示（可选）

### 阶段 8 · 安全点展示（1 分钟，可选加分项）
- 演示未登录访问 `/admin/users` → 被强制跳登录
- 演示学生账号直接构造请求访问 `/api/users`（用浏览器 DevTools 在 Console 试 `fetch('/api/users')`） → 返回 401/403
- 演示连续密码错误 5 次 → 触发 1 分钟锁定

## 注意事项

1. **演示环境请用 `init_database.sql` 初始化**——否则可能因脏数据导致数据不一致
2. **演示「权限越权拦截」时**，用 Postman 或浏览器 DevTools 手动构造请求效果最好：
   - 复制学生登录拿到的 Token
   - 用它请求 `POST /api/users` → 预期返回 `code=403`
3. **演示审核流程时**，如果觉得「先发布再审核」太耗时，可以预先在数据库里准备 1~2 条 `audit_status=0` 的数据
4. **图片上传功能**：`uploads/` 目录会在项目运行目录下自动创建；清理时注意删除演示临时图片
5. **清理日志功能**：`DELETE /api/admin/logs/clean?daysBefore=30` 清理的是 30 天以前的日志；刚初始化的数据全是当天的，清理结果会是「清理了 0 条」，**这是正常的**（可以在答辩时解释「演示用数据太新，真实场景下每天凌晨 2 点会自动清理」）
6. **验证码**：每次刷新页面或点击验证码图片会生成新的随机字符；为了演示便利，前端验证码不强制匹配（仅作视觉展示，后端不校验）—— 正式生产环境需要把验证码移到后端
7. **密码加密传输**：登录/注册时前端用 XOR+Base64 加密（`encrypt.js`），后端解密后再做 BCrypt 比较 —— 目的是避免明文密码在网络中被抓包看到；生产环境建议直接上 HTTPS
